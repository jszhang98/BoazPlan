# UX Research Plan — BoazPlan (pilot)

**Date:** 2026-02-13
**Author:** UX Designer (via BMAD)
**Scope:** Validate core flows (New Request creation, Privacy/Approval, Volunteer assignment) and measure friction that affects the PRD success metrics (median submit time ≤ 2 minutes; request response rate ≥ 70% within 72 hours).

## Objectives
- Validate that users can complete the **New Request** flow within the 2-minute target and identify friction points.
- Test pastor/admin approval and privacy controls for clarity and trust.
- Validate volunteer assignment flow and basic notification behavior from a volunteer perspective.
- Surface UI copy, discoverability, and privacy concerns that might reduce adoption.

## Success criteria (what 'good' looks like)
- Median task completion time (prototype) ≤ 2 minutes for primary path (≥ 80% of participants).
- ≥ 80% of pastors understand approval settings and can change visibility correctly in the prototype.
- Volunteers can accept/decline an assignment and update status with ≤ 1 error per task.
- Qualitative: No major privacy / trust blockers reported by pastors or members.

## Target participants (recruitment profiles)
- Members (primary): 6–8 — typical church member, smartphone user (mobile-first).  
- Pastors/Admins: 3–4 — decision-makers who will use approval workflows.  
- Volunteers / Group leaders: 3–4 — frequently-assigned users who accept/complete tasks.

## Method & sessions
- Method: Moderated remote usability tests + short follow-up interviews.  
- Session length: 45–60 minutes (task run + interview).  
- Prototype: Clickable mobile PWA prototype (New Request flow, approval queue, volunteer accept flow, notifications).  
- Platform: Zoom / Teams + screen share + usability recording (consent required).

## Tasks & script (moderated)
1. Warm-up (5 min): Confirm role, experience with church apps, device used.
2. Task A — Create a New Request (primary success path): "Post a help request for a neighbor needing a ride, set visibility, and submit." (Measure time & errors.)
3. Task B — Edit / Save Draft (edge case): "Start a request, simulate a dropped connection, then resume and submit."
4. Task C — Pastor Approval Flow: "As Pastor, find pending requests, review one, and set visibility to 'Group only' with pastor approval required."  
5. Task D — Volunteer assignment & update: "As volunteer, accept an assignment and mark it in-progress and then complete it."  
6. Debrief interview (15 min): Perceived trust/privacy, confusing steps, suggestions, likelihood-to-use.

## Metrics to capture
- Quantitative: Task completion time, success/failure rate, # of errors, time to recover from offline, SUS-like quick rating (1–5).  
- Qualitative: Pain points, privacy concerns, confusion, suggested improvements.

## Sample size & timeline
- Sample: 12–16 participants total (6–8 members, 3–4 pastors, 3–4 volunteers).  
- Timeline: 2 weeks (recruit 3–4 days, test 3–5 days, analyze 2–3 days).  
- Deliverables within 10 business days: research report, prioritized findings, annotated screenshots, recommended UX fixes.

## Recruitment & incentives
- Recruit via partner churches / local networks; include tech-savvy and less-tech-savvy members.  
- Incentive: $25–$50 gift card or equivalent per participant.

## Risks & mitigations
- Risk: Prototype fidelity too low → mitigate with clear task scripting and set expectations.  
- Risk: Sensitive privacy concerns → use anonymized scenarios and obtain consent for recording.

## Deliverables
- Usability findings report (includes measured median times vs 2-minute target).  
- Prioritized list of UX changes (quick wins + medium changes).  
- Updated prototype recommendations and copy changes.

## Next steps (if plan approved)
1. Build clickable prototype for the three core flows.  
2. Recruit participants and schedule sessions.  
3. Run moderated sessions and synthesize findings.  
4. Produce report and update PRD + Epics/Stories for fixes.
