# Story 2.1: Minimal Request creation flow

Status: ready-for-dev

## Story

As a Member,
I want a short multi-field flow to create a request,
so that I can submit within 2 minutes and get help quickly.

## Acceptance Criteria

1. Mobile-first request creation UI with fields: type, short description, visibility (public/private/group), urgency, optional contact.
2. Median submit time ≤ 2 minutes in usability check for pilot users.
3. Draft is auto-saved locally during entry (IndexedDB) and restored on restart.
4. Successful POST to `POST /requests` returns 201 with request id and initial status `pending`.
5. If visibility requires pastor approval, request appears in approval queue and PII is hidden when flagged anonymous.
6. Analytics events emitted: `request.start`, `request.draft.save`, `request.submit` and `request.success`.
7. WCAG AA basics met for the flow (labels, focus order, contrast, screen-reader announcements for errors & success).

## Dev Notes

- Key UX constraints: single-screen flow, minimize fields to meet 2-minute target. See PRD: "Request creation UI (short, mobile-optimized flow) + 2-minute target".
- Backend: keep request create idempotent-friendly (support `Idempotency-Key` header).
- Offline: store drafts in IndexedDB and queue failed POSTs for retry-on-reconnect (see Offline epic tasks).
- Analytics: fire `request.start` on open; `request.draft.save` on autosave; `request.submit` on submit; `request.success` on 201.

### Project structure hints

- DB model: Request { id, type, description, visibility, urgency, anonymous, status, createdBy, churchId, createdAt }
