# 教会内部互助系统项目简介

## 一、项目概述

建立一个以教会为基本单位的内部互助系统，旨在通过数字工具实现教会内部资源的爱心共享、代祷守望与实际生活协助，增强肢体关怀与服侍效率。

## 二、目标

- 促进物资与服务在会众内部的高效流转与共享
- 提供代祷与灵性支持的线上记录与通知机制
- 支持实际生活帮助（接送、陪诊、家访、短期住宿等）任务分配与追踪
- 保护隐私与由教会治理的审核流程

## 三、目标用户

- 教会会众（成员）
- 牧者与事工领袖（审批、调度）
- 志愿者与小组负责人成员

## 四、核心功能（MVP）

- 1. 用户与教会单位注册/邀请、角色与权限管理
- 2. 互助发布：物资、服务、求助与代祷申请表单
- 3. 配对与响应：志愿者报名、领任务与状态更新
- 4. 代祷墙：请求、更新、代祷回应与定期提醒
- 5. 审核流程：牧者/小组长审批与可见范围控制
- 6. 基本通知：应用内通知与邮件/短信（可选）
- 7. 活动/任务日历与短期日程管理

## 五、非功能性需求

- 数据隐私：个人信息最小化存储与教会可控的数据导出/删除
- 可用性：移动优先响应式界面，低带宽适配
- 安全性：认证、授权、审计日志与简单的备份方案
- 可维护性：模块化设计，便于教会自定义字段与工作流程

## 六、成功衡量指标

- 月活跃用户（MAU）与每月互助请求数量
- 请求响应率（被志愿者接单的占比）
- 牧者/管理员满意度与重复使用率

## 七、建议技术选型（可选）

- 前端：React 或 Vue（移动适配、PWA）
- 后端：Node.js + Express / NestJS 或 Spring Boot（Java）
- 数据库：Postgres 或 Firebase（取决于实时需求）
- 身份：教会邮箱/手机号 + OTP 或 SSO（若教会已有系统）
- 部署：云主机 + 定期备份，或使用托管平台（Firebase/Heroku）

## 八、实现路线（三阶段）

- 阶段 0 - 发现与治理：访谈牧者与小组，确定隐私与审批策略（2周）
- 阶段 1 - MVP（核心互助与代祷功能）：用户/教会注册、发布/响应、审批、通知（6-8周）
- 阶段 2 - 扩展（日历、统计、导出、移动优化、离线支持）：集成短信/邮件、优化体验（4-6周）

## 九、风险与缓解

- 风险：隐私与误用 → 缓解：严格审批、最小化公开信息、审计日志
- 风险：志愿者不足 → 缓解：自动提醒、激励机制、小组竞争/荣誉榜
- 风险：技术维护负担 → 缓解：选择托管服务并提供管理员培训
